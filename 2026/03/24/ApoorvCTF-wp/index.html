具体利用

程序中是存在后门函数 magic 的。所以这里的具体利用思路就是将某个 note 节点的 print_note 函数指针给修改掉。

而具体修改的操作只有在 add_note 的时候书写content的时候能做到，所以进一步的思路：释放某个note后，在随后申请新的note，这个新的note对应的content指针指向的堆恰好是原先note的head部分（即0x10的堆头部分）

具体利用思路：

  1. 申请一个0x10的note0，struct note_head 0x8； content 0x10
  2. 申请一个0x10的note1，struct note_head 0x8； content 0x10
  3. 将note0删除，将note1也删除，这两个note的note_head都会进入 0x10的fastbin，content会进入0x18的fastbin
  4. 新申请一个0x8的note2，会优先从fastbin中申请堆块，根据LIFO原则，note1的struct note_head会变成note2的struct note_head，note0的note_head会变成note2的content指针指向堆块，所以就能对note2的content进行修改，进而修改了note0的note_head，note0的note_head包含 print_note 的地址，能够修改为 magic 的地址。
  5. 调用 index = 0 对应的 print_note 函数，就调用了 magic 函数。

在 did_i_pass 函数中，会对一个程序最开始的一个全局变量（也是一个堆）进行值检查，如果满足特定的值，就能够拿到flag了。这个全局变量是程序申请的第一个堆：

可以看到，每个堆块的大小都是0x30。

现在就是需要利用UAF漏洞，来修改第一个堆块的值为check值。本题用的glibc版本是2.35

glibc 2.32 之前，tcache 和 fastbin 的 free chunk 里存的 fd 指针就是明文的下一个 chunk 地址。攻击者一旦拿到 UAF 写，直接把 fd 改成任意地址，malloc 就返回那个地址，实现任意写。太容易了。所以 glibc 2.32 引入了 safe-linking（提出者是 Check Point Research），核心思路是：存储时做一次异或混淆，取出时再异或还原。

所以我们可以先申请两个 order ，然后将他们free掉，这样在相同大小的 tcache bin 中，链表结构如下：

order1 → order0 → NULL

所以 order0 的 fd值就是 0 ^ (pos >> 12) = pos >> 12

这里的pos指的就是堆块对应user data的起始地址。右移12位等价于页对齐，所以这个其实再左移12位就得到了堆地址。可以通过 inspect_order 进行泄漏，从而计算出 did_i_pass 功能函数中对应第一个堆块的 user data 。然后通过 tcache poisoning 对target的值进行修改。

tcache 是 LIFO（后进先出），本质就是一个单链表栈。每个 size class 有一个链表头和一个 count 计数器。核心伪代码：

具体利用流程：

  1. 申请两个order，然后释放掉，得到 order1 → order0 → NULL 的tcache bin单向链表。
  2. inspect_order 泄露heap_addr，计算出 heap base 地址。
  3. 链表目前是: order1 → order0 → NULL，count = 2，所以修改 order1 的fd为 target addr ^ heap key。这里的heap key 就是之前泄露的 heap_addr
  4. 修改后，链表为： order1 → target_chunk。
  5. 然后申请两个堆块，对应 order[2] 和 order[3] ，这里的order[3]就是target_chunk。
  6. 通过 modify_order 对target_chunk的user data进行修改即可。

Abyss

这个题目按理我独立完成的话会比较困难，因为程序功能比较多，也比较复杂，感觉逆向功底不够。但是发现里面有 io_uring 相关的内容，这个模块是Linux近两年出现的，用于异步I/O通信。

程序是实现了一个自定义命令行的功能，用help查看存在的命令有：

Commands: DIVE DESCEND WRITE POP FLUSH STATUS BEACON ABYSS ECHO HELP QUIT，在IDA里查看一下每个对应的功能

整体看完这个程序，觉得需要好好逆向理解一番（Claude居然短短5分钟就完成了我大概两个小时的工作量）

新开一个线程，线程以调用 start_routine() 来开启，并以 arg 为唯一参数传递

在这个程序中，整体有三个线程并发进行：

  1. main 2. benthic_thread(void *) 3. mesopelagic_thread(void *)

我们先来看 main 线程：

里面有四个全局注册表：

  1. g_dive_reg[16] — 最多16个 dive 对象指针
  2. g_levi_reg[24] — 最多24个 leviathan 对象指针
  3. g_request_stack — 无锁链表栈（所有活跃 dive 对象）
  4. g_slab_secret — 8字节随机密钥，用于 XOR 加密指针

g_slab_secret 是由系统调用 getrandom 生成的8字节随机数。

dive对象和levi对象的结构体在IDA中恢复如下:

先开启了另外两个线程，并且通过 pipe 管道来进行信息交换。main函数里可以接受的命令有:

  • DIVE [depth]，从dive slab池里取一个槽出来，深度为depth，注册到 g_dive_reg 中，并且压入 g_request_stack 中
  • DESCEND [dive id] [depth] [hexdata]，向 dive 对象 [+offset+8] 写入 hex 解码数据，offset ∈ [0, 63]，最大写入 64-offset 字节，具体解码用了 hexdecode 函数，
  • WRITE [id] [offset] [hexdata]，和DESCEND差不多，但是这次是向 dive 对象 [+8] 写入hex解码数据。
  • POP，从 g_request_stack 中弹出一个dive_node，然后从 g_dive_reg 中删掉，并且将magic改为 deadbeef 。并且打印出 dive_node 的地址，应该是bss段和 g_request_stack 相关的地址，可以用来计算 PIE base 。
  • FLUSH，通知 mesopelagic 线程将 g_request_stack 中的所有slab node回收。
  • STATUS [id]，输出 g_dive_reg[i] 的所有信息。
  • BEACON [id]，从levi slab池里取一个槽出来，levi node和dive node 的结构差不多，都记录在上面。 levi slab池如果空了，就会从 dive slab 池中借用（这里感觉比较奇怪，先记录一下），注册到 g_levi_reg 中。
  • ABYSS [id]，会将device id传递 benthic_thread 线程，将 leviathan 的 data 作为 io_uring SQE 提交给 benthic 线程执行
  • ECHO [text]，回显文本
  • QUIT，退出程序
  • HELP，显示程序包含什么命令

mesopelagic 线程从 main 线程通过 g_cmd_pipe 读取内容，如果是1的话，则执行FLASH将 g_request_stack 全部清空，如果读到0则直接返回。

看完整个程序之后，我并没有发现什么很明显的洞，不过这么多线程，并且加入了 io_uring 这一子模块，猜测是与之相关的机制漏洞。

需要具体看一下程序相关的沙箱:

Seccomp 沙箱策略

  • Main Thread: 允许常规 I/O 和网络，禁止 execve/fork/openat
  • Benthic Thread: 允许 openat + io_uring 全套（setup/enter/register），禁止网络和 execve
  • Mesopelagic Thread: 允许 read/write/nanosleep，禁止 execve/openat

可以看到，上面的反汇编代码中，确实将 g_request_stack 中的所有活跃节点都恢复到不活跃状态了，但是对应的 g_dive_reg 注册表中的指针并没有置为null，这里存在UAF漏洞。当节点都恢复到不活跃状态后，执行 WRITE 和 DESCEND 的时候就可以向已经释放的空间写入内容。
