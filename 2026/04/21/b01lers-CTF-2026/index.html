题目背景

这个题目的程序实现了一个轻量化的 forth 解释器。

forth解释器是什么：

Forth解释器是一种基于堆栈的交互式系统，它通过“内部”和“外部”两个层次解释并执行以空格分隔的命令（字，words）。它不仅是解释器，也是编译器和汇编器，能动态操作数据栈和字典，以其极其简洁、紧凑和在嵌入式系统中的直接硬件控制能力而闻名。

Reddit

核心组成与工作原理

  • 双层结构：

    • 外部解释器：读取用户输入的文本行，识别命令是数值、已定义的字还是需要编译的新字。
    • 内部解释器：执行已编译的Forth字序列（执行流），实现快速的执行效率。

  • 数据栈 (Data Stack)：Forth的核心，所有操作数（数字）都先入栈，操作（字）直接在栈上进行计算。

  • 字典 (Dictionary)：Forth系统由字（word）组成，所有定义过的函数均以字典形式存储。

    知乎专栏

特点与应用

  • 即时编译：输入的命令立刻在内存中即时编译并执行，无需耗时的外部链接过程。
  • 轻量化：很多Forth解释器仅需几KB内存，非常适合低功耗设备和微控制器。
  • 交互式开发：支持在交互式提示符下即时定义和测试新功能。

利用步骤

利用流程可以总结成四步：

  1. 是造 UAF。先定义 foo，再定义 foo_parent 引用 foo，这样 foo_parent 的 compile_def 里会直接保存 foo->word_t 的地址，不是保存字符串 “foo”。然后 forget foo，源码会把 foo->word_t 和 foo->param 都 free 掉，但 foo_parent 里那个指针不会被修复，于是得到悬空指针。
  2. 是把悬空指针先接管成一个合法的 docol 对象。我们再定义一个新词 fake_word_t，利用 0x30 tcache 的复用，让 fake_word_t->word_t 恰好落回之前 foo->word_t 的地址。这样 foo_parent 的悬空指针虽然还是原地址，但实际已经指向了 fake_word_t。你已经验证过这一点：执行 foo_parent . 能打印新词里的常量。之所以选 docol，是因为用户定义词除了 word_t 之外，还自带一个单独分配的 param，也就是 compile_def，这块通常是 0x90 chunk，后面还能继续利用。
  3. 是把这个 docol 对象再次释放，扩展成“两块可利用的悬空资源”。执行 forget fake_word_t 后，会同时 free 掉两个东西： fake_word_t->word_t 的 0x30 chunk，以及 fake_word_t->param 的 0x90 chunk。接着我们用一个长名字的新词去重占那块 0x90，把它布置成命令字符串，比如 cat</flag;#AAAA…；再用一个 30 字节左右的名字去重占那块 0x30，把悬空 word_t 的 code 字段改成程序启动时泄露出来的 dosys 地址。这里利用的是 word_t 结构里 code 在偏移 0x18，param 在偏移 0x20，而 tcache free 后前 16 字节会被覆盖，但后面的 code/param 常常还能保留或被部分覆盖。
  4. 是触发。此时 foo_parent 里还保留着那个旧地址，所以执行 foo_parent 时，程序实际会把这个悬空对象当作一个 word_t 来调用，也就是执行 word->code(word->param)。如果 code 已经被我们改成 dosys，而 param 又指向我们布置好的命令字符串，就会走到的 system(val)，从而执行 /bin/sh 拿到 shell。

为什么这条链成立

word_t 布局是：

关键点是：

  • free() 进 tcache 后，前 16 字节会被 tcache 元数据覆盖
  • 但 code 和 param 这两个字段通常还留着
  • 所以如果你先把悬空对象变成一个合法的 docol 词，再 free 它，它的：
    • code = docol
    • param = 某个 0x90 chunk往往还在，但是也被 free 掉了

然后你只要把这个 freed word_t 再拿回来，用可控字节覆盖到 code，把code修改为 dosys 的地址，而 param 继续沿用原来的那个 0x90 地址就行。

然后申请一个名字很长的命令，来占用这个 0x90 的chunk，这个命令就是我们 dosys 对应的 param ，里面放一个 /bin/sh 或者是 cat /flag 这种就可以拿到shell或者flag了。

程序做了一个最小堆结构实现的优先队列，保护情况：

有功能：

  1. insert
  2. delete
  3. peek
  4. edit
  5. count
  6. quit

主要漏洞点有以下几处：

  1. insert的时候，堆块大小由输入数据动态决定，而edit的时候可以修改堆的大小为 0x20，这里存在堆溢出

flag所处的堆并不在优先队列中，第一步要想办法泄露 heap_base_addr，通过堆溢出就可以达到了。问题是后面该怎么拿到一个堆的指针能够指向flag所在的chunk，或者是分配一个堆到flag chunk之前呢？

刚开始最直接的想法就是用 fastbin attack。但是溢出长度不太够，最多就覆盖掉chunk header字段，没办法写到 fd 指针处。

中间调试尝试的过程忘了记了，直接给出利用思路：

  1. 先做第一次扩容并拿到可编辑的堆顶

    插入 8 个 “a”，再插入 “!”。

    这样第 9 个消息会触发 array 扩容，而且 “!” 作为最新的chunk，不仅会成为最小堆顶，后续还与 存array指针的chunk 响铃

  2. 利用 edit() 对堆顶小 chunk 的 32 字节写，做越界读泄漏 heap

    对 “!” 对应的小 chunk 写入 32 字节，覆盖掉它自己的结尾并踩到后面 array1 的 chunk header。然后 peek()，puts(array[0]) 会一路越界读到 array1 用户区里的堆指针，从而泄漏 heap 地址。

    再由固定偏移算出 heap_base 和 flag_chunk_addr。

  3. 恢复现场，避免后续 delete() 出问题，然后再次 edit()，把当前根 chunk 修回一个正常短字符串，同时把后面被踩坏的 header 恢复掉。

  4. 重新布置堆，让最后存活的根 chunk 紧挨 array2

    再插入 7 个 “b” 和 1 个 “c”，然后连续 delete() 16 次。

    目标是让最后剩下的根节点变成那个内容为 “c” 的 chunk，并且它物理上正好挨着第二次扩容后的 array2。

  5. refill 到第二次扩容边界，再伪造 array2->size，具体操作就是：连续插入 31 个 “d”，把队列重新填满到触发第三次扩容前的状态。这时利用根 chunk “c” 的 edit()，越界改掉后面 array2 的 size，伪造成一个更大的 chunk，比如 0x521。

  6. 触发第三次扩容，由于glibc的机制，会计算当前chunk + chunk_size的的大小与 top chunk 的位置进行比较。这里由于改了 chunk_size，所以系统认为该 chunk 与 top chunk 响铃，所以 free(array2) 就把 top chunk给前移了。插入 “e”。在 insert(“e”) 里会先分配 “e”，再分配新的 array3，最后 free(old array2)。因为已经伪造了 array2->size，free(array2) 不会正常入 bin，而是直接和 top 合并，把 top 前移到 old array2 附近。

  7. 从被前移的 top 连续切小块，做到和 live array3 overlap

    连续插入 24 个 “~”。

    这些小分配会从新的 top 中不断切出 0x20 小 chunk，最终走到 array3 的用户区。

  8. 插入一个新的 “!”，让 overlap chunk 成为堆顶这样就能再次通过 edit()，直接写到 array3 指针数组。

  9. 覆写 array3[0] 和 array3[1] 为 flag_chunk_addr，在 overlap chunk 上 edit()，保住 array3 的 header，同时把前两个指针改成 flag_chunk_addr。

    这样逻辑堆顶对应的字符串指针就变成了 flag 所在堆地址。

  10. peek() 直接打印 flag

    因为 peek() 本质上是 puts(array[0])，而此时 array[0] == flag_chunk_addr，所以会直接输出 flag。
